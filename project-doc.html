Chapter 1

What Did We Build?

A personal portfolio website with a live backend — not just a static HTML page.

This project is a full-stack personal portfolio. Visitors can see your skills, projects, certifications, experience, and blog posts. They can also send you a message directly from the page and it arrives in your Gmail inbox in real time.

Behind the scenes, an Express.js server runs on the cloud, connects to a MongoDB database, and handles API requests from the browser.

Simple analogy Think of it like a restaurant: the frontend (HTML/CSS/JS) is the dining area customers see. The backend (Node.js + Express) is the kitchen that processes orders. The database (MongoDB) is the pantry/storage. Render is the building that hosts the whole restaurant.

Key Features

Night Sky Hero
3-layer parallax star animation, nebula orbs, mountain silhouettes — all pure CSS + JS canvas.
Dynamic Blog
Write and publish posts from an admin panel. Posts are stored in MongoDB and loaded live.
Contact Form with Email
Visitor fills a form → backend sends a formatted email to anukool.xeep@gmail.com instantly.
Resume Download
Server streams the PDF file directly. No exposing the file URL publicly.
Dark / Light Mode
CSS variables swap the whole theme. Preference is saved to localStorage.
Animated Skill Bars
Progress bars fill with a shimmer effect when scrolled into view, using IntersectionObserver.

Example 1: Visitor loads the blog section

  1. 1

    Browser visits https://anukoolyadav.onrender.com → Render receives the request → server.js serves public/index.html (static file).

  2. 2

    main.js runs in the browser, reaches the Blog section, calls fetch('/api/posts').

  3. 3

    Express matches the URL to postRoutes.js → requireDB middleware checks MongoDB is ready → postController.getAllPublished() runs.

  4. 4

    Mongoose queries MongoDB: Post.find({ published: true }) → returns array of posts as JSON.

  5. 5

    Response travels back to the browser → main.js renders each post as a card on the page.

Example 2: Visitor submits the contact form

  1. 1

    Visitor fills Name, Email, Message → clicks Send → browser calls fetch('/api/contact', { method: 'POST', body: JSON }).

  2. 2

    Express matches to contactRoutes.js → contactController.sendMessage() runs.

  3. 3

    Controller validates the data (no empty fields, valid email format).

  4. 4

    Nodemailer connects to Gmail SMTP using the App Password → sends formatted HTML email to anukool.xeep@gmail.com.

  5. 5

    Controller responds { success: true } → browser shows green "Message sent!" banner.

Example 3: Admin creates a new blog post

  1. 1

    Admin opens /admin.html, enters password → browser stores it in memory.

  2. 2

    Admin writes title + content in Markdown, clicks Publish → browser calls POST /api/posts with header x-admin-pass: anukool.

  3. 3

    auth.js middleware reads the header → matches ADMIN_PASS env variable → allows through.

  4. 4

    postController.create() runs → Mongoose validates the Post schema → saves to MongoDB.

  5. 5

    Next visitor who loads the site sees the new post automatically.

Chapter 5

Backend Code — File by File

What each backend file contains and why it's written that way.

server.js — Entry Point

This is the first file Node runs. Its only job is to connect everything together.

require('dotenv').config();            // load .env file first
const express = require('express');
const path    = require('path');
const { connectDB } = require('./src/config/db');

const app  = express();
app.use(express.json());
app.use(express.static(path.join(__dirname, 'public')));  // serve frontend files

app.get('/health', (req, res) => res.send('OK'));       // Render health check
connectDB();                                               // connect to MongoDB

app.use('/api/posts',   require('./src/routes/postRoutes'));
app.use('/api/admin',   require('./src/routes/adminRoutes'));
app.use('/api/contact', require('./src/routes/contactRoutes'));

app.listen(PORT, '0.0.0.0', () => console.log('Server running'));

src/config/db.js — Database Connection

Connects to MongoDB and retries every 10 seconds if it fails (so the server never crashes just because the DB is slow).

let dbReady = false;

function connectDB() {
  mongoose.connect(uri, { serverSelectionTimeoutMS: 8000 })
    .then(() => { dbReady = true; })          // success → mark ready
    .catch(err => setTimeout(connectDB, 10000)); // fail  → retry in 10s
}

module.exports = { connectDB, isReady: () => dbReady };

src/models/Post.js — Data Schema

A schema is a blueprint. It tells MongoDB: "every blog post must have these exact fields with these types."

const postSchema = new mongoose.Schema({
  title:     { type: String, required: true },   // mandatory
  content:   { type: String, required: true },
  excerpt:   { type: String },                    // auto-generated summary
  tags:      [String],                            // array of strings
  published: { type: Boolean, default: true },
  createdAt: { type: Date,    default: Date.now }
});

src/middleware/auth.js — Password Guard

Middleware runs between the route matching and the controller. It can block a request before anything else runs.

module.exports = (req, res, next) => {
  if (req.headers['x-admin-pass'] === process.env.ADMIN_PASS)
    return next();   // password correct → continue
  res.status(401).json({ error: 'Unauthorized' });
};

src/routes/postRoutes.js — URL Mapping

Routes connect an HTTP method + URL pattern to a controller function. Middleware is added in between where needed.

router.get('/',      requireDB,             post.getAllPublished);  // public
router.get('/:id',   requireDB,             post.getOne);
router.post('/',     auth, requireDB,       post.create);          // needs password
router.put('/:id',   auth, requireDB,       post.update);
router.delete('/:id', auth, requireDB,      post.remove);

Frontend Code — Key Concepts

How the visual magic in the browser works.

CSS Variables (Custom Properties)

Instead of hardcoding colours everywhere, we define them once at the top. Changing the theme is as easy as changing one value.

:root {
  --headline: #4ade80;   /* bright green for headings */
  --teal:     #4ecdc4;   /* buttons, focus glow */
  --card:     #0a1628;   /* card background */
}

/* Light mode just overrides the same variables */
.light :root {
  --headline: #16a34a;   /* darker green for white background */
}

IntersectionObserver — Scroll Animations

This browser API fires a callback when an element enters the screen. No timers or scroll event listeners needed — it's efficient.

const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      entry.target.classList.add('visible');  // triggers CSS animation
      observer.unobserve(entry.target);        // only animate once
    }
  });
});

document.querySelectorAll('.reveal').forEach(el => observer.observe(el));

Fetch API — Talking to the Backend

JavaScript calls the Express API without reloading the page.

// GET: load blog posts
const posts = await fetch('/api/posts').then(r => r.json());

// POST: submit contact form
const res = await fetch('/api/contact', {
  method:  'POST',
  headers: { 'Content-Type': 'application/json' },
  body:    JSON.stringify({ name, email, message })
});
const json = await res.json();   // { success: true } or { error: '...' }

Parallax Stars

Three layers of stars (far, mid, near) move at different speeds when you scroll. This creates the illusion of depth.

// Far stars: move slowly (small parallax factor)
farStars.style.transform = `translateY(${scrollY * 0.1}px)`;
// Near stars: move faster (larger parallax factor)
nearStars.style.transform = `translateY(${scrollY * 0.4}px)`;

Secrets & Environment Variables

How sensitive information (passwords, API keys) is handled safely.

NEVER put passwords in your source code or commit them to GitHub. If someone forks your repo, they get your credentials. Environment variables are the safe alternative.

The .env File (Local Only)

This file lives on your machine only. It is listed in .gitignore so it is never uploaded.

# .env — local development secrets
MONGODB_URI=mongodb+srv://Anukool7:Anu%401234@cluster0...
ADMIN_PASS=anukool
PORT=3000
EMAIL_USER=anukool.xeep@gmail.com
EMAIL_PASS=abcdefghijklmnop   ← 16-char Gmail App Password

How dotenv Works

require('dotenv').config();   // reads .env, loads into process.env

// Now accessible anywhere in Node.js:
process.env.MONGODB_URI   // the connection string
process.env.EMAIL_PASS    // the Gmail app password

On Render (Production)

Render has an "Environment" tab in your service dashboard. You type the same keys and values there. When your server starts on the cloud, Render injects them — no .env file needed.

render.yaml — Auto-configuration

Instead of clicking buttons in Render's UI, we tell Render exactly how to run the app in a file:

services:
  - type: web
    name: anukool-portfolio
    runtime: node
    buildCommand: npm install       # installs packages
    startCommand: npm start         # runs node server.js
    healthCheckPath: /health        # Render pings this to confirm server is up

Health Check

Render periodically calls GET /health. If it gets back 200 OK, the server is healthy. If not, Render marks the deployment failed. Our server always responds to this regardless of MongoDB state.

MongoDB Atlas Connection

  • Create a free cluster at cloud.mongodb.com
  • Go to Network Access → Add IP Address → 0.0.0.0/0 (allow all — Render uses dynamic IPs)
  • Go to Database Access → Create a user with password
  • Click Connect → Drivers → copy the URI
  • Replace @ in password with %40 (URL encoding for special characters)
